Observed issue: steady FPS degradation over time
GPU utilization climbs from ~60% to 95%+ over the course of the simulation
causing FPS to drop from 70 to 36 (worse with sloshing: down to 24).
The degradation is permanent per session — FPS never recovers.
Memory usage is mostly flat, so this is not a memory leak, as can be checked like so:
nvidia-smi --query-gpu=memory.used,utilization.gpu --format=csv -l 1
Disabling the D3D12 debug layer did not resolve it.

This was caused by spatial cache coherence loss. At startup, particles are laid out
in a regular grid in memory so spatial neihgbors have nearby buffer indices. When 
the shaders iterate over the 27 neighboring grid cells and read particle[j] for
each neighbor, those j values are often contiguous, and so there are many cache hits.
As the simulation runs, however, particles drift in space, and move physically but
their buffer indices stay fixed, so the buffer indices of spatial neighbors start becoming
random across the particle buffer. Each particle[j] read is a cache miss essentially,
so the GPU stalls while the particle's data is fetched from the VRAM. The fix is to
periodically rearrange the particle buffer so that particles in the same grid cell
are contiguous in memory again. This is effectively a counting sort keyed by grid cell index.

Observed issue: particles teleporting to bottom-left corner (NaN bug)

Some particles spontaneously jumped to the -x, -z corner of the bounding box and "boiled"
there. The symptom worsened when particles were compressed into a corner for an extended
period. The cause turned out to be NaN appearing in particle positions -- HLSL's
clamp(NaN, boxMin, boxMax) returns boxMin per IEEE 754, which explains the bottom-left
teleport.

Finding the source of the NaN was difficult because the HLSL compiler aggressively optimizes
away isnan() checks. When the compiler can reason via data-flow analysis that a value "can't"
be NaN from the surrounding arithmetic, it eliminates the check entirely. This meant that
isnan()-based instrumentation consistently reported zero NaN occurrences even though the output
was clearly NaN. Bit-level checks via asuint() were more reliable.

The root cause was overlapJitter() in SphKernels.hlsli. When two particles overlap (distance
near zero), the spiky gradient returns zero and they can never separate, so deltaCS uses a
pseudo-random jitter direction to nudge them apart. The original implementation computed this
direction as normalize(float3(sin(...), sin(...), sin(...))), where the sin arguments are
linear combinations of the two particle indices scaled by large primes. For large particle
indices (thousands), these arguments reach tens of thousands, where GPU sin() loses precision.
For specific (i,j) pairs, all three sin components landed near zero simultaneously, and
normalize(~zero) computed ~0 * rsqrt(~0) = ~0 * INF = NaN.

The fix was to guard the normalization with a length check: if the squared length of the
vector is below a threshold (1e-12), fall back to a fixed arbitrary unit direction instead
of trying to normalize a near-zero vector.

since then we've moved on to a different jitter implementation that doesn't use sin() at all

Resource state validation error after adding the readback buffer

After adding the particle readback buffer (for reading back density data to the CPU), the D3D12
debug layer started reporting an INVALID_SUBRESOURCE_STATE error every frame. The error fired
during particleMesh->Draw(), specifically when SetGraphicsRootDescriptorTable bound the SRV
descriptor that points at the particle buffer. The debug layer complained that the buffer was
not in the correct state for SRV access.

The particle buffer has two roles in the pipeline: the compute shaders write to it as a
RWStructuredBuffer (UAV), and the vertex shader reads from it as a StructuredBuffer (SRV). In
DX12, you have to explicitly transition resources between these roles using barriers, because
the GPU needs to handle caching differently for writes vs reads.

The thing is, this was technically wrong even before the readback buffer was added -- the buffer
was in UAV state when GraphicsPass ran, and the vertex shader was reading it as an SRV without
a proper transition. The reason the debug layer didn't catch it before is that the only barriers
on the particle buffer during the frame were UAV execution barriers (between compute dispatches),
not Transition barriers. Without any explicit state transitions on the resource, the debug layer
uses lenient implicit-promotion rules and doesn't flag the mismatch.

The readback copy introduced two explicit Transition barriers (UAV -> COPY_SOURCE -> UAV) on the
particle buffer. Once the debug layer sees you explicitly managing a resource's state with
transitions, it switches to strict tracking for that resource -- it knows exactly what state the
buffer is in at every point and validates every access against it. So when the particle mesh
tried to bind the SRV while the buffer was in UAV state, it got caught.

The fix was to add explicit transitions around GraphicsPass: UAV -> SRV before drawing, then
SRV -> UAV after, so the buffer is in the correct state when the vertex shader reads it. The
target state has to be NON_PIXEL_SHADER_RESOURCE | PIXEL_SHADER_RESOURCE (both bits), because
the root signature's default DATA_STATIC_WHILE_SET_AT_EXECUTE descriptor volatility makes the
debug layer require the full 0xC0 mask regardless of which shader stage actually reads it.
